\begin{abstract}
This report studies the design and empirical behavior of MAPPO and HAPPO on two cooperative StarCraft Multi-Agent Challenge (SMAC) maps: \texttt{3s5z} and \texttt{8m\_vs\_9m}. Both algorithms are on-policy actor-critic methods trained under centralized training and decentralized execution. MAPPO uses a shared PPO-style actor update with a centralized critic, while HAPPO updates agent policies sequentially and corrects later updates with a factor that accounts for earlier policy changes. I implement the comparison in HARL, run smoke and pilot checks, and then train a single-seed full matrix for 20 million environment steps per completed run. On \texttt{3s5z}, MAPPO and HAPPO both reach a best win rate of 1.0. On \texttt{8m\_vs\_9m}, MAPPO obtains the higher final win rate, while the HAPPO recovery run reaches the higher best win rate but shows stronger late-stage fluctuation.
\end{abstract}

\section{Introduction}

The StarCraft Multi-Agent Challenge (SMAC) evaluates cooperative control in partially observable micromanagement tasks \cite{samvelyan2019starcraft}. Each allied unit is controlled by one agent. During execution, an agent observes only local unit features and legal actions; during training, algorithms may use global state information to learn better value functions. This setting naturally tests centralized training with decentralized execution (CTDE), because good performance requires both individual action selection and team-level coordination.

This report focuses on MAPPO and HAPPO. MAPPO extends PPO to cooperative multi-agent tasks and is a strong practical baseline when implementation details are tuned carefully \cite{yu2022surprising}. HAPPO is designed for heterogeneous-agent policy optimization and updates agents sequentially with a correction factor, which gives a more structured multi-agent policy improvement step \cite{zhong2024heterogeneous}. Rather than only reporting final numbers, the report explains the algorithm design choices, how those choices are realized in HARL, and how the experiment matrix is controlled.

\section{Algorithm Design}

\subsection{Problem Formulation}

SMAC can be modeled as a cooperative partially observable Markov game. At step \(t\), each agent \(i\) receives a local observation \(o_i^t\), selects a discrete action \(a_i^t\), and shares a team reward \(r^t\). The joint action is \(\mathbf{a}^t=(a_1^t,\ldots,a_n^t)\). The training objective is to maximize the expected discounted return
\[
J(\theta)=\mathbb{E}_{\pi_\theta}\left[\sum_{t=0}^{T}\gamma^t r^t\right],
\]
where \(\gamma\) is the discount factor. In the implementation used here, agents execute recurrent policies and the critic can use centralized information, matching the CTDE design.

\subsection{MAPPO Design}

MAPPO keeps PPO's clipped policy-gradient update and adapts it to a multi-agent rollout. The actor maps each agent's local observation and recurrent hidden state to an action distribution. The critic estimates a centralized value function from global information, which reduces variance in advantage estimation while preserving decentralized execution.

For an action sampled under the old policy, the PPO ratio is
\[
r_t(\theta)=\frac{\pi_\theta(a_t\mid o_t)}{\pi_{\theta_{\text{old}}}(a_t\mid o_t)}.
\]
The clipped actor objective is
\[
L^{\text{MAPPO}}(\theta)=
\mathbb{E}\left[\min\left(r_t(\theta)A_t,\,
\text{clip}(r_t(\theta),1-\epsilon,1+\epsilon)A_t\right)\right].
\]
Advantages are computed with generalized advantage estimation using \(\gamma=0.95\) and \(\lambda=0.95\). The training design also includes entropy regularization to avoid premature deterministic policies, gradient clipping for stable updates, recurrent actors for partial observability, and value normalization for critic learning.

In this experiment, MAPPO is implemented with parameter sharing because the agents in the selected SMAC maps are homogeneous within each team. The selected HARL commit does not provide tuned MAPPO configs for \texttt{3s5z} and \texttt{8m\_vs\_9m}, so I generate MAPPO configs from the same-map HAPPO tuned configs and switch the algorithm-specific settings to the MAPPO shared-actor design. This keeps map-level rollout, evaluation, and network settings comparable across algorithms while acknowledging that MAPPO is not using a separately tuned map-specific config.

\subsection{HAPPO Design}

HAPPO changes the actor update from a simultaneous shared update to a sequential agent-wise update. Suppose agents are updated in an order \(1,\ldots,n\). When updating agent \(i\), previous agents may already have changed their policies. HAPPO therefore multiplies the surrogate objective by a factor \(M_i\) that summarizes the effect of earlier policy-ratio changes. A simplified form is
\begin{equation}
\begin{aligned}
L_i^{\text{HAPPO}}(\theta_i)
= \mathbb{E}\Big[
& M_i \min\big(
r_{i,t}(\theta_i) A_t, \\
& \mathrm{clip}(r_{i,t}(\theta_i), 1-\epsilon, 1+\epsilon) A_t
\big)
\Big]
\end{aligned}
\end{equation}
After one agent is updated, the factor is updated with the ratio between its new and old action probabilities, and the next agent uses the updated factor. This design is more faithful to the joint-policy nature of multi-agent optimization than simply applying independent PPO updates to each agent.

The HAPPO configs use separate actor parameters rather than MAPPO-style sharing. This is more expensive, but it matches HAPPO's design goal of handling heterogeneous policies. On the selected SMAC maps the agents are mostly homogeneous, so HAPPO's advantage is not guaranteed by the task structure; the experiment is therefore useful for observing whether the sequential update improves or destabilizes learning under the same training horizon.

\section{Environment Installation Process}

The installation process was a non-trivial part of the experiment because SMAC depends on both Python packages and the native StarCraft II Linux binary. I first created an isolated Conda environment for HARL instead of using the system Python, then installed HARL in editable mode, installed PyTorch with CUDA support, and installed the StarCraft Multi-Agent Challenge package from the oxwhirl repository. This repository choice matters: the package name \texttt{smac} on PyPI can refer to SMAC3, which is unrelated to the StarCraft benchmark. Installing the wrong package produces import and environment errors even though the package name looks correct.

After the Python environment was prepared, I installed StarCraft II Linux 4.10, copied the SMAC map files into the StarCraft II map directory, and verified that the map registry could list \texttt{3s5z} and \texttt{8m\_vs\_9m}. I then ran a random-agent SMAC example to confirm that the SC2 binary, PySC2 controller, maps, and Python environment could work together. A second practical issue was NumPy compatibility: the HARL code used deprecated aliases such as \texttt{np.int} and \texttt{np.bool}, while the local environment used NumPy 2. I patched these aliases before running experiments. Table~\ref{tab:install} summarizes the installation checks that mattered for the final runs.

\section{Experimental Design}

\subsection{Maps and Agents}

The experiment uses two SMAC maps with different difficulty profiles. The \texttt{3s5z} map contains a mixed allied group of stalkers and zealots, testing coordination among unit types with different combat roles. The \texttt{8m\_vs\_9m} map is an asymmetric marine battle where eight allied marines must defeat nine enemy marines, making focus fire and damage trading important. The second map is harder because the allied team starts with a numerical disadvantage.

\subsection{Training Matrix}

The full matrix is MAPPO/HAPPO \(\times\) \texttt{3s5z}/\texttt{8m\_vs\_9m}. For each combination, I first run a short smoke test to ensure that StarCraft II, SMAC, HARL, logging, and plotting work. I then run a 10,000-step pilot to catch configuration errors. The final comparison uses a single seed and 20 million environment steps for each completed full run. Thus a 20 million-step run produces 250 evaluation rows. I report both final win rate and best win rate because final performance reflects the state at the end of training, while best win rate reveals whether a method learned a strong policy but later became unstable.

\subsection{Recovery Policy for Failed Runs}

The original HAPPO run on \texttt{8m\_vs\_9m} stopped producing new progress after 5.12 million steps. To avoid overwriting data, I kept that run as a stale checkpoint and launched a new recovery run with the same map, algorithm, and seed but a different experiment prefix. The recovery run reached 20 million environment steps and is used as the full HAPPO result for \texttt{8m\_vs\_9m}. The stale checkpoint remains in the result table only to make the data lineage clear.

\section{Analysis}

On \texttt{3s5z}, both algorithms learn strong policies. MAPPO finishes with a final win rate of 0.975, while HAPPO finishes at 0.875. Both reach a best evaluation win rate of 1.0. This indicates that the mixed-unit map can be solved by both update designs under the selected training budget. MAPPO's higher final value suggests that the shared update is stable on this homogeneous cooperative task.

On \texttt{8m\_vs\_9m}, MAPPO reaches a final win rate of 0.875 and a best win rate of 0.9. HAPPO recovery reaches a higher best win rate of 0.975 but ends at 0.75. This pattern suggests that HAPPO did find a very strong policy during training, but its late-stage performance was less stable. One possible reason is that sequential per-agent updates introduce extra sensitivity to update order and advantage estimates; another is that the asymmetric marine map amplifies small policy changes because losing one allied marine early can quickly change the battle outcome.

The comparison should therefore be read as a single-seed behavioral study rather than a final ranking. MAPPO is more stable by final win rate in this run, while HAPPO shows a stronger peak but larger fluctuation. This is exactly why both final and best win rate are reported.

\section{Observations and Reflections}

My first observation is that environment correctness is easy to underestimate in SMAC. A Python import check alone is not enough: the SC2 binary, game version, map files, PySC2 controller, and SMAC package must all match. The most useful engineering strategy was to build the experiment in layers: first a random-agent check, then smoke tests, then 10,000-step pilot runs, and only then full 20 million-step training. This prevented long runs from hiding simple configuration errors.

My second observation is that final win rate and best win rate tell different stories. On \texttt{8m\_vs\_9m}, HAPPO recovery achieved the best win rate of 0.975, which means it did discover a strong policy. However, its final win rate was only 0.75, so the learned policy was not as stable at the end. If I only reported the final value, I would miss the peak-performance behavior; if I only reported the best value, I would hide the instability. Reporting both metrics gives a more honest picture.

My third reflection concerns algorithm design. MAPPO is simpler because parameter sharing turns a homogeneous team into a large shared-policy dataset, and this seems to help stability on the selected SMAC maps. HAPPO has a more careful sequential update rule, but that rule also makes the update depend on agent order and on the quality of the correction factor. In a single-seed run, this can show up as stronger oscillation even when the method reaches a high peak win rate. Therefore, I would not conclude that HAPPO is worse; I would conclude that HAPPO needs multi-seed evaluation and possibly more tuning on \texttt{8m\_vs\_9m}.

Finally, the stale original HAPPO run changed how I think about experimental bookkeeping. Long MARL experiments can fail silently by stopping progress output while the surrounding session still appears alive. Keeping the stale run, launching a separate recovery run, and recording both paths made the result easier to audit. This is a practical lesson: reproducible MARL work depends not only on algorithms, but also on conservative data management.

\section{Limitations}

The main limitation is the single-seed design. SMAC training is noisy, and a stronger claim would require multiple seeds with mean and standard deviation. A second limitation is configuration parity: HAPPO uses official HARL tuned configs, while MAPPO uses generated configs derived from the same-map HAPPO settings because tuned MAPPO configs for the selected maps were unavailable in the chosen HARL commit. Finally, the analysis uses evaluation reward and win rate only; replay-level behavior, loss curves, entropy, and value estimates would provide a more detailed view of why a run becomes stable or unstable.

\section{Conclusion}

This report compares MAPPO and HAPPO from both a design and experimental perspective. MAPPO uses a shared PPO-style actor update with a centralized critic, while HAPPO applies sequential agent updates with a correction factor. In the single-seed SMAC experiments, both algorithms solve \texttt{3s5z}; on \texttt{8m\_vs\_9m}, MAPPO has the better final win rate, whereas HAPPO reaches the better best win rate but fluctuates more near the end of training. The results support the practical value of MAPPO as a stable baseline and show that HAPPO's sequential design can reach strong policies but may require more careful multi-seed evaluation.
