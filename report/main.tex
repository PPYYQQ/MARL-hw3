\begin{abstract}
This report reproduces MAPPO and HAPPO on the StarCraft Multi-Agent Challenge (SMAC) using the HARL codebase. The study covers environment setup, implementation mapping, smoke tests, pilot runs, result collection, win-rate visualization, and a single-seed full training matrix on \texttt{3s5z} and \texttt{8m\_vs\_9m}. All four MAPPO/HAPPO by map combinations were run to 20 million environment steps. The original HAPPO run on \texttt{8m\_vs\_9m} stopped producing new progress after 5.12 million steps, so a non-destructive recovery run was launched and used as the completed full-run result for that matrix entry. The final results show strong performance from both algorithms on \texttt{3s5z}; on \texttt{8m\_vs\_9m}, MAPPO has the higher final win rate, while the HAPPO recovery run reaches a higher best win rate but shows more late-stage fluctuation.
\end{abstract}

\section{Introduction}

The StarCraft Multi-Agent Challenge (SMAC) is a cooperative multi-agent benchmark built on StarCraft II micromanagement tasks \cite{samvelyan2019starcraft}. Each allied unit is controlled by one agent with partial local observations, while the team objective is to defeat the enemy units. This setting is useful for studying decentralized execution, centralized training, credit assignment, and stability in multi-agent reinforcement learning.

This homework focuses on two on-policy methods: MAPPO and HAPPO. MAPPO adapts PPO to cooperative multi-agent tasks and has been shown to be a strong baseline when implementation details are carefully tuned \cite{yu2022surprising}. HAPPO extends policy optimization to heterogeneous-agent settings through sequential policy updates and a factor correction term \cite{zhong2024heterogeneous}. The goal of this project is not only to run the algorithms, but also to connect the paper-level ideas with the actual HARL implementation, produce reproducible scripts, and analyze the training results on two SMAC maps.

\section{Algorithms}

\subsection{MAPPO}

MAPPO can be viewed as a multi-agent version of PPO. Each agent acts with a decentralized policy, while training uses centralized information to estimate the value function. The actor update still follows the clipped PPO surrogate objective, which limits overly large policy updates by clipping the probability ratio. In cooperative SMAC tasks, the key practical ingredients are stable advantage estimation, centralized critics, entropy regularization, and careful control of rollout and evaluation settings.

In this project, MAPPO configurations were generated from the corresponding HAPPO tuned configurations because the selected HARL commit did not include tuned MAPPO configuration files for \texttt{3s5z} and \texttt{8m\_vs\_9m}. The generated MAPPO configurations switch the algorithm-specific options to parameter sharing and fixed agent order, while keeping map-level training parameters aligned with the HARL tuned settings.

\subsection{HAPPO}

HAPPO introduces sequential policy updates for heterogeneous agents. During one update cycle, agents are updated one by one rather than all at once. A factor term tracks how earlier policy updates affect the joint policy probability ratio, and this factor is included in the later agents' surrogate objectives. This design gives HAPPO a more principled monotonic-improvement structure than a naive independent PPO-style update.

For SMAC, the selected HAPPO runs use the official HARL tuned configurations. The final matrix uses a recovery run for HAPPO on \texttt{8m\_vs\_9m}, because the original full run became stale after reaching 5.12 million environment steps. The stale original output is preserved for auditability, while the completed recovery run is used for the final comparison.

The main difference between HAPPO and MAPPO appears at the runner and actor-update levels. HAPPO initializes the factor to one, updates agents sequentially, writes the current factor into the actor buffer, and then updates the factor with the new-to-old action-probability ratio. MAPPO also uses the PPO clipped surrogate, but it does not apply the HAPPO sequential factor. Under parameter sharing, MAPPO merges data across agents to update a shared actor.

\section{Environment and Reproducibility}

Experiments were run on a Linux workstation with an NVIDIA GeForce RTX 5090 GPU. The Python environment uses Conda, Python 3.10, PyTorch with CUDA support, HARL at commit \texttt{b1af98b0dbab72a2eee9d160751cd09aedbb8ce2}, and the oxwhirl SMAC package. StarCraft II Linux 4.10 and SMAC maps were installed locally, and the target maps \texttt{3s5z} and \texttt{8m\_vs\_9m} were verified through SMAC map-list and random-agent checks.

Two setup issues were addressed during the project. First, the PyPI package named \texttt{smac} installs SMAC3 rather than the StarCraft Multi-Agent Challenge environment, so the setup script installs SMAC from the oxwhirl repository. Second, the local PyTorch and NumPy combination required using NumPy 2, so a small compatibility patch replaces old HARL uses of \texttt{np.int} and \texttt{np.bool} with built-in Python types.

All scripts, configuration snapshots, raw \texttt{progress.txt} files, processed summaries, figures, and validation logs are included in the repository. The progress tracker records each major change and its Git commit.

\section{Experimental Protocol}

The target matrix contains MAPPO and HAPPO on \texttt{3s5z} and \texttt{8m\_vs\_9m}. The workflow has four levels:

\begin{enumerate}
  \item \textbf{Dry run}: print the four experiment commands.
  \item \textbf{Smoke test}: run very small experiments to verify that SC2, SMAC, HARL, logging, and plotting work end to end.
  \item \textbf{Pilot run}: run each matrix entry for 10,000 environment steps to catch configuration errors before full training.
  \item \textbf{Full run}: use HARL tuned training length, 20 million environment steps, for the final single-seed matrix.
\end{enumerate}

HARL writes evaluation rows to \texttt{progress.txt}. The collection script scans the HARL output tree and the saved raw-result tree, deduplicates rows by map, algorithm, experiment name, run path, and step, and writes a unified CSV. A separate summarization script produces a Markdown table with final and best evaluation metrics, and the plotting script generates win-rate curves for each map.

\section{Analysis}

On \texttt{3s5z}, both algorithms learn strong policies. MAPPO finishes with a higher final win rate of 0.975, while HAPPO finishes at 0.875. Both reach a best evaluation win rate of 1.0, which suggests that the map is solvable by both methods under the selected single-seed setup.

On \texttt{8m\_vs\_9m}, the final comparison is more nuanced. MAPPO reaches a final win rate of 0.875 and a best win rate of 0.9. The HAPPO recovery run reaches a higher best win rate of 0.975 but ends at 0.75, indicating larger late-stage fluctuation. This does not prove that one algorithm is generally better than the other, because the experiment uses only one seed and one generated MAPPO configuration per map. It does show that the pipeline is capable of reproducing full-length training curves and exposing stability differences across runs.

The stale original HAPPO run on \texttt{8m\_vs\_9m} is an important engineering detail. Instead of deleting or overwriting it, the project preserves the original directory and launches a recovery run with a new experiment prefix. This keeps the data auditable while still completing the missing matrix entry.

\section{Limitations}

The main limitation is that the final matrix uses a single random seed. SMAC training can be noisy, especially on asymmetric maps such as \texttt{8m\_vs\_9m}, so a stronger conclusion would require multiple seeds and reporting means, standard deviations, and confidence intervals. Another limitation is configuration parity: the selected HARL commit does not provide tuned MAPPO configurations for the two target maps, so MAPPO uses generated configurations derived from the HAPPO tuned settings. This is practical for a homework reproduction, but it may differ from a fully tuned MAPPO baseline.

The report also focuses on evaluation win rate and average episode reward from HARL's \texttt{progress.txt}. A deeper study could include loss curves, entropy, value estimates, replay inspection, and qualitative analysis of unit-control behavior.

\section{Conclusion}

This project completes an end-to-end reproduction workflow for MAPPO and HAPPO on two SMAC maps. The repository contains environment scripts, configuration snapshots, raw and processed results, plots, validation logs, and an Overleaf-ready LaTeX project. The experiments show that both algorithms can solve \texttt{3s5z} under the selected setup, while \texttt{8m\_vs\_9m} exhibits more visible stability differences. The remaining work for a publication-grade comparison would be to run multiple seeds and report aggregate statistics.
